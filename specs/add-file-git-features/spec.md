## User Scenarios & Testing *(mandatory)*

### User Story 1 - 浏览当前项目文件 (Priority: P1)

作为使用 `cc-pet-web` 的开发者，我希望在应用内看到当前连接绑定工作区的文件与目录结构，以便不离开当前界面就能定位该连接对应项目的文件、查看内容，并理解项目当前有哪些可操作文件。

**Why this priority**: 文件列表是后续文件操作和 Git 变更定位的入口；当前项目只有聊天附件展示和消息搜索，没有工程文件浏览能力。

**Acceptance Scenarios**:

1. **[US1-1]**
   **Given** 用户打开文件列表入口，且当前连接已从配置文件绑定可访问的项目工作区  
   **When** 文件列表加载完成  
   **Then** 用户应看到以该连接绑定工作区为根的文件和目录列表，并能区分文件、目录及常见文件类型

2. **[US1-2]**
   **Given** 文件列表中存在多层目录  
   **When** 用户展开或折叠某个目录  
   **Then** 系统应只展示用户当前展开范围内的子项，并保持当前选中文件或目录的上下文可识别

3. **[US1-3]**
   **Given** 文件列表中存在可查看文件  
   **When** 用户选择该文件  
   **Then** 系统应展示文件内容或可读预览，并明确显示当前查看的文件名称和路径

4. **[US1-4]**
   **Given** 用户从一个连接切换到另一个已配置工作区的连接  
   **When** 文件列表重新加载  
   **Then** 系统应展示新连接绑定工作区的文件结构，而不是继续展示上一个连接的文件结构

**Edge Cases**:

- **[US1-5]** 当目录为空时，系统应显示空状态，而不是让用户误以为加载失败
- **[US1-6]** 当某个文件或目录无法访问时，系统应提示该项不可访问，并允许用户继续浏览其他文件
- **[US1-7]** 当文件体积过大或无法安全预览时，系统应说明无法直接预览，并避免阻塞整个文件列表
- **[US1-8]** 当当前连接没有配置工作区或配置路径无效时，系统应提示需要先配置有效工作区

### User Story 2 - 管理当前项目文件与目录 (Priority: P1)

作为开发者，我希望能在文件列表中创建、重命名和删除当前连接绑定工作区内的文件或目录，以便完成轻量项目整理和 AI 生成结果后的人工修正。

**Why this priority**: 用户明确选择首版需要完整文件操作能力；该能力让文件列表从只读索引变成可完成基础项目管理的工作面板。

**Acceptance Scenarios**:

1. **[US2-1]**
   **Given** 用户位于当前连接绑定工作区的某个可写目录  
   **When** 用户创建新文件或新目录并提供名称  
   **Then** 系统应在目标目录中新增对应项目，并在文件列表中立即展示新增结果

2. **[US2-2]**
   **Given** 用户选择一个可重命名的文件或目录  
   **When** 用户输入新名称并确认  
   **Then** 系统应完成重命名，并更新文件列表中该项目的名称和路径展示

3. **[US2-3]**
   **Given** 用户选择一个可删除的文件或目录  
   **When** 用户确认删除操作  
   **Then** 系统应删除该项目，并从文件列表中移除它

4. **[US2-4]**
   **Given** 用户正在查看一个可编辑文本文件  
   **When** 用户修改内容并保存  
   **Then** 系统应保存文件内容，并向用户反馈保存结果

**Edge Cases**:

- **[US2-5]** 当用户提供空名称、重复名称或包含非法路径字符的名称时，系统应阻止操作并说明原因
- **[US2-6]** 删除非空目录前，系统应给出明确确认，避免用户误删大量内容
- **[US2-7]** 当文件在用户操作期间被外部进程修改、删除或重命名时，系统应提示当前列表已过期，并允许用户刷新后继续
- **[US2-8]** 所有文件操作必须限制在当前连接绑定工作区内；试图越过工作区边界的操作必须被拒绝

### User Story 3 - 查看 Git 变更与 Diff (Priority: P1)

作为开发者，我希望看到当前连接绑定工作区的 Git 变更列表，并查看单个文件的差异内容，以便快速理解 AI 或人工改动的影响范围。

**Why this priority**: 用户明确选择首版 Git 范围为状态列表和 diff 查看；这也是参考 Nezha 的 Git 体验中最小可用、风险较低的闭环。

**Acceptance Scenarios**:

1. **[US3-1]**
   **Given** 当前连接绑定工作区是 Git 仓库，且存在已修改、新增、删除或重命名的文件  
   **When** 用户打开 Git 变更入口  
   **Then** 系统应展示变更文件列表，并为每个文件标明其变更状态

2. **[US3-2]**
   **Given** Git 变更列表中存在可比较的文件  
   **When** 用户选择某个变更文件  
   **Then** 系统应展示该文件的 diff 内容，并清晰区分新增、删除和上下文内容

3. **[US3-3]**
   **Given** 用户正在文件列表中浏览文件  
   **When** 某个文件存在 Git 变更  
   **Then** 系统应在文件列表中提供可识别的变更提示，帮助用户从文件树定位变更文件

4. **[US3-4]**
   **Given** 工作区没有未提交变更  
   **When** 用户打开 Git 变更入口  
   **Then** 系统应显示无变更状态，并保持刷新入口可用

**Edge Cases**:

- **[US3-5]** 当当前连接绑定工作区不是 Git 仓库或 Git 信息不可用时，系统应说明 Git 状态不可用，而文件浏览功能仍可继续使用
- **[US3-6]** 当 diff 内容过大或包含二进制文件时，系统应给出摘要或不可预览提示，避免页面卡死
- **[US3-7]** 当文件状态在用户查看期间变化时，系统应允许用户刷新 Git 状态并更新 diff

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须从当前连接的配置文件中获取该连接绑定的工作区，并提供该工作区的文件与目录列表，以层级结构呈现。
- **FR-002**: 用户必须能够展开、折叠目录，并选择文件查看内容或可读预览。
- **FR-003**: 系统必须在文件列表中区分文件、目录、常见文件类型以及不可访问项。
- **FR-004**: 用户必须能够在当前连接绑定工作区内创建文件和目录。
- **FR-005**: 用户必须能够在当前连接绑定工作区内重命名文件和目录。
- **FR-006**: 用户必须能够在当前连接绑定工作区内删除文件和目录，且删除前必须有明确确认。
- **FR-007**: 用户必须能够查看并保存可编辑文本文件的内容。
- **FR-008**: 系统必须阻止任何越过当前连接绑定工作区边界的文件查看或文件操作。
- **FR-009**: 系统必须提供当前连接绑定工作区的 Git 变更列表，并展示每个变更文件的状态。
- **FR-010**: 用户必须能够从 Git 变更列表中选择文件并查看该文件的 diff。
- **FR-011**: 文件列表必须能提示哪些文件存在 Git 变更，便于用户从目录结构中定位改动。
- **FR-012**: 系统必须为文件列表、文件操作、Git 状态和 diff 提供刷新或重新加载能力。
- **FR-013**: 当文件或 Git 信息不可用时，系统必须提供明确错误或空状态，并允许用户继续使用不受影响的功能。

### Non-Functional Requirements *(if applicable)*

- **NFR-001**: 文件列表和 Git 变更列表在常见项目规模下应保持可交互，不应因单个大文件、大目录或大 diff 阻塞整体界面。
- **NFR-002**: 文件操作必须具备明确反馈，包括成功、失败、无权限、冲突和取消状态。
- **NFR-003**: 危险操作必须可识别、可取消，并避免误删当前连接绑定工作区之外的任何内容。
- **NFR-004**: Git 状态展示应足够及时，支持用户在外部工具修改文件后重新同步当前视图。

### Key Entities *(if applicable)*

- **连接工作区**: 每个连接从配置文件读取到的项目根目录，是该连接下所有文件浏览、文件操作和 Git 查看能力的边界。
- **文件项**: 工作区中的文件或目录，包含名称、路径、类型、访问状态和可操作状态。
- **文件内容**: 用户选择文件后看到或编辑的内容；可能是文本、Markdown、图片或无法预览的文件。
- **Git 变更**: 当前连接绑定工作区中相对 Git 基线发生变化的文件记录，包含路径和变更状态。
- **Diff**: 某个 Git 变更文件的差异视图，用于说明新增、删除和上下文内容。

### Dependencies and Assumptions

- 当前 Spec 的工作区由每个连接的配置文件决定；用户切换连接时，文件列表和 Git 变更应随当前连接切换到对应工作区。
- 当前项目已有聊天附件和消息搜索能力，但它们不是工程文件列表；本功能应作为新的项目文件/Git 工作面板看待。
- `nezha` 的参考价值主要是产品体验层面：完整文件树、Git 状态提示、文件查看编辑、Git 变更列表和 diff 查看；本 Spec 不要求复刻其完整 IDE、任务系统或桌面能力。

- **BM-002**: 用户从发现 AI 产生改动到打开对应 diff 的平均时间下降。

## Out of Scope *(if applicable)*

明确不在本次功能范围内的内容：

- 运行时任意选择本机目录、手动添加项目工作区、或脱离连接配置的多项目管理。
- Git stage、unstage、commit、AI commit message、分支创建/切换/合并/删除、push、pull、提交历史浏览。
- 与远程 Git 托管平台集成，例如 Pull Request、Issue、CI 状态或代码评审。
- 完整 IDE 能力，例如语言服务、自动补全、重构、调试、终端集成。
- 聊天附件系统和历史消息搜索的合并改造。
